# Развёртывание и эксплуатация WaterSysPro

## 1. Обзор

Этот документ описывает полный цикл развёртывания и эксплуатации системы WaterSysPro:

- подготовка аппаратной части и прошивки ESP32;
- развёртывание инфраструктуры (MQTT‑брокер, backend, база данных);
- загрузка Web‑интерфейса и работа с microSD;
- подключение мобильного приложения;
- базовые операции эксплуатации и диагностики.

Для детальной информации по отдельным компонентам см.:

- прошивка ESP32: [`firmware_overview.md`](docs/firmware_overview.md)
- аппаратная часть: [`hardware_overview.md`](docs/hardware_overview.md)
- backend: [`backend_overview.md`](docs/backend_overview.md)
- мобильное приложение: [`mobile_overview.md`](docs/mobile_overview.md)
- логирование и SD: [`firmware_logging.md`](docs/firmware_logging.md), [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md)


## 2. Подготовка аппаратной части

1. Собрать аппаратную часть в соответствии с рекомендациями из [`hardware_overview.md`](docs/hardware_overview.md):
   - подключить ESP32, реле, датчики уровня, дисплей, энкодер;
   - установить и подключить модуль microSD;
   - обеспечить раздельное питание логики и силовой части.
2. Проверить правильность подключения по пинам, заданным в конфигурации прошивки.
3. Убедиться в наличии защитных автоматов, заземления и соблюдении норм электробезопасности.


## 3. Сборка и прошивка ESP32

### 3.1 Настройка окружения

1. Установить VS Code и PlatformIO.
2. Открыть проект WaterSysPro (корень: `d:/WatersysPro`).
3. Убедиться в корректности конфигурации [`platformio.ini`](platformio.ini):
   - выбранная плата (например, `esp32dev`);
   - зависимости `lib_deps`;
   - отключение SPIFFS (`-DUSE_SPIFFS=0`), если используется SD‑только конфигурация.

### 3.2 Сборка и загрузка

1. Собрать прошивку:
   - через интерфейс PlatformIO в VS Code или командой `pio run`.
2. Подключить ESP32 по USB и загрузить прошивку:
   - `pio run -t upload`.
3. Открыть Serial‑монитор:
   - `pio device monitor`.
4. Проверить вывод инициализации:
   - сообщение о старте системы;
   - результаты инициализации SD, RTC, WiFi (если включено);
   - отсутствие критических ошибок.

Чек‑лист smoke‑тестов и дополнительные комментарии см. в [`REFRACTOR_AND_SMOKE_SUMMARY.md`](docs/REFRACTOR_AND_SMOKE_SUMMARY.md).


## 4. Подготовка microSD и Web‑интерфейса

### 4.1 Загрузка файлов UI

1. Отформатировать microSD‑карту (FAT32).
2. Скопировать статические файлы Web‑интерфейса из каталога [`закинуть на сд`](закинуть на сд) на корень SD:
   - `index.html`
   - `logfilter.html`
   - при необходимости — дополнительные файлы (CSS, JS, иконки).

В проекте может быть скрипт [`upload_sd.py`](upload_sd.py), упрощающий загрузку файлов на SD‑карту. Используйте его при необходимости.

### 4.2 Установка SD в устройство

1. Вставить подготовленную SD‑карту в слот на плате.
2. Перезапустить устройство.
3. В Serial‑мониторе убедиться, что SD успешно смонтирована (см. логи `initSD()` из [`utils.cpp`](src/utils.cpp)).
4. Проверить, что Web‑сервер отдаёт `index.html` по запросу на корневой URL (`/`).

Подробности интеграции SD: [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md).


## 5. Развёртывание backend и MQTT инфраструктуры

### 5.1 Подготовка окружения

1. Установить Docker и Docker Compose на сервере или ПК, где будет работать backend.
2. Скопировать репозиторий WaterSysPro на сервер.
3. Перейти в корень проекта, где расположен [`docker-compose.yml`](docker-compose.yml).

### 5.2 Настройка переменных окружения

1. В каталоге [`backend`](backend) скопировать файл `.env.example` в `.env` и заполнить параметры:
   - параметры БД (PostgreSQL);
   - параметры MQTT;
   - параметры HTTP‑сервера;
   - секреты аутентификации.
2. При необходимости скорректировать порты и имена контейнеров в [`docker-compose.yml`](docker-compose.yml).

### 5.3 Запуск контейнеров

1. Выполнить:

```bash
docker-compose up -d
```

2. Дождаться, пока поднимутся контейнеры:
   - `mosquitto` — брокер MQTT;
   - `db` (или аналогичное имя) — PostgreSQL;
   - `backend` — Node.js backend;
   - (опционально) `nginx` — обратный прокси.

3. Проверить работоспособность backend:
   - открыть в браузере `http://<host>:<port>/api/health`;
   - убедиться, что возвращается корректный JSON с признаком `ok`.

Подробнее о backend см. [`backend_overview.md`](docs/backend_overview.md) и [`backend/README.md`](backend/README.md).


## 6. Подключение ESP32 к WiFi и backend

### 6.1 Настройка WiFi

1. Подключиться к локальному Web‑интерфейсу устройства (по умолчанию через WiFi‑AP, который поднимает ESP32 при отсутствии настроек):
   - SSID вида `WSystem-XXXX`;
   - IP по умолчанию `192.168.4.1`.
2. Открыть в браузере `http://192.168.4.1/`.
3. Ввести admin‑token по умолчанию (см. [`config.h`](include/config.h)) и перейти к настройке WiFi.
4. Через Web‑форму или эндпоинт `POST /api/wifi` задать:
   - `ssid` и `password` для основной сети;
   - язык интерфейса (опционально);
   - при необходимости — другие параметры.

После перезагрузки устройство попытается подключиться к указанной сети и получить IP по DHCP.

### 6.2 Настройка MQTT и связи с backend

1. В инженерном меню или через Web‑API (`/api/mqtt`) задать параметры MQTT:
   - адрес брокера (hostname/IP контейнера `mosquitto`);
   - порт (обычно 1883);
   - логин/пароль (если используются);
   - base‑topic (должен соответствовать настройке в backend).
2. Убедиться по логам (Serial и backend), что:
   - ESP32 успешно подключился к MQTT‑брокеру;
   - публикует статус‑сообщения;
   - backend получает их и обновляет состояние устройств.


## 7. Подключение и настройка мобильного приложения

1. Настроить и запустить backend и MQTT, как описано выше.
2. В каталоге [`mobile`](mobile) установить зависимости и запустить Expo (см. [`mobile_overview.md`](docs/mobile_overview.md)):
   - `npm install`
   - `npm start` или `expo start`
3. В конфигурации мобильного приложения (например, в `app.json` или экране настроек) указать адрес backend:
   - `http://<host>:<backend_port>`.
4. Запустить приложение на смартфоне (через Expo Go или собранный APK/AAB).
5. Войти в приложение (если используется аутентификация), выбрать устройство и убедиться, что:
   - отображается статус устройства;
   - команды управления (включение реле и т.п.) доходят до ESP32 через backend и MQTT.


## 8. Ежедневная эксплуатация

### 8.1 Основные действия оператора

- мониторинг состояния через:
  - локальный дисплей и меню;
  - Web‑интерфейс ESP32;
  - мобильное приложение;
- управление режимами:
  - запуск/остановка технологического цикла;
  - запуск обратной промывки при необходимости;
- проверка логов и аварий:
  - просмотр логов на дисплее (через меню);
  - загрузка лог‑файлов с SD через Web‑интерфейс;
  - анализ логов в случае аварий.

### 8.2 Регламентные работы

- периодическая проверка и очистка логов (через инженерное меню или Web‑интерфейс);
- проверка состояния SD‑карты (свободное место, отсутствие ошибок);
- проверка состояния датчиков уровня (отсутствие загрязнений, корректная калибровка);
- проверка работы насосов и реле (отсутствие клина контактов, перегрева);
- обновление прошивки ESP32 и Web‑интерфейса при выходе новых версий.


## 9. Обновление прошивки и Web‑интерфейса

### 9.1 Обновление прошивки ESP32

1. Собрать новую версию прошивки через PlatformIO.
2. Загрузить её на устройство (`pio run -t upload`).
3. Убедиться по логам, что все подсистемы инициализировались корректно.
4. Проверить работу FSM, датчиков, реле и Web‑интерфейса.

При существенных изменениях структуры логов или настроек сверяться с [`REFRACTOR_AND_SMOKE_SUMMARY.md`](docs/REFRACTOR_AND_SMOKE_SUMMARY.md).

### 9.2 Обновление Web‑интерфейса

1. Обновить файлы в каталоге [`закинуть на сд`](закинуть на сд) (HTML/JS/CSS) до новой версии.
2. Залить их на SD‑карту:
   - либо вручную через кардридер;
   - либо с помощью скрипта [`upload_sd.py`](upload_sd.py).
3. Вставить SD‑карту в устройство и перезапустить его.
4. Проверить корректность отображения и работы нового Web‑интерфейса.


## 10. Диагностика и устранение неполадок

### 10.1 Типичные проблемы и проверки

1. **Устройство не поднимает Web‑интерфейс:**
   - проверка логов `initSD()` (наличие и состояние SD‑карты);
   - проверка наличия файлов `index.html` и других ресурсов на SD;
   - проверка работы WiFi (STA/AP), IP‑адреса и подключения клиента.

2. **Нет связи с backend/MQTT:**
   - проверка MQTT‑настроек на ESP32 и в backend;
   - проверка доступности брокера (порт 1883, контейнер mosquitto);
   - просмотр логов backend и брокера.

3. **Неверные уровни в баках:**
   - проверка установок датчиков, наличия помех и препятствий;
   - проверка и, при необходимости, корректировка калибровки уровней в инженерном меню.

4. **Частые аварии по сухому ходу или таймаутам:**
   - проверка исправности насосов и гидравлики;
   - проверка корректности настроек `SafetySettings` (таймауты стадий, параметры сухого хода);
   - анализ логов для выявления закономерностей.

### 10.2 Использование логов для диагностики

1. Просматривать логи на дисплее через инженерное меню (см. [`firmware_logging.md`](docs/firmware_logging.md)).
2. Выгружать логи на SD (экспорт) и анализировать их на ПК.
3. При необходимости подключаться по Serial и просматривать отладочные сообщения `logDebug`, `logInfo` и др. из [`utils.cpp`](src/utils.cpp).
