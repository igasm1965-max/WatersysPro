# Backend WaterSysPro: архитектура и API

## 1. Назначение backend

Backend WaterSysPro — это серверное приложение, которое обеспечивает:

- приём телеметрии от контроллеров WaterSysPro через MQTT или HTTP;
- хранение данных о устройствах и их состоянии в базе данных;
- единый HTTP API для мобильного приложения и других внешних клиентов;
- буферизацию и маршрутизацию команд управления от клиентов к устройствам.

Backend реализован на Node.js и разворачивается вместе с брокером MQTT и базой данных через Docker Compose. Основные файлы backend находятся в каталоге [`backend`](backend).


## 2. Архитектура backend

### 2.1 Основные компоненты

Структура каталога [`backend`](backend):

- точка входа HTTP‑сервера и маршрутизация API: [`server.js`](backend/src/server.js)
- модуль работы с базой данных: [`db.js`](backend/src/db.js)
- вспомогательные инструменты: [`src/tools`](backend/src/tools)
- статические файлы и прошивки: [`static`](backend/static)
- конфигурация Docker и Nginx: [`Dockerfile`](backend/Dockerfile), [`nginx.conf.example`](backend/nginx.conf.example)
- описание API и запуска: [`README.md`](backend/README.md)

Backend, как правило, разворачивается внутри Docker‑контейнера и взаимодействует с:

- брокером MQTT (Mosquitto), запущенным в контейнере `mosquitto` (см. [`docker-compose.yml`](docker-compose.yml));
- базой данных (PostgreSQL), запущенной в отдельном контейнере.

### 2.2 Поток данных

Типовой поток данных выглядит так:

1. **ESP32 → MQTT → backend:**
   - контроллер публикует телеметрию (состояние реле, уровни баков, аварии, служебные события) в заданный MQTT‑топик;
   - backend подписывается на эти топики и при получении сообщения:
     - разбирает JSON‑payload;
     - обновляет состояние устройства в базе данных;
     - при необходимости генерирует события для клиентов (WebSocket, push‑уведомления).

2. **Клиент → HTTP API → backend → MQTT → ESP32:**
   - мобильное приложение или веб‑клиент отправляет команду управления на HTTP‑эндпоинт backend;
   - backend проверяет авторизацию, валидирует команду и публикует соответствующее сообщение в MQTT‑топик устройства;
   - ESP32 подписан на этот топик и выполняет команду (переключение реле, изменение настроек), а затем публикует новый статус.

3. **Backend → HTTP API → клиент:**
   - клиенты могут запрашивать текущий статус устройства и историю событий;
   - backend берёт данные из БД и/или кеша и отдаёт в формате JSON.


## 3. HTTP API backend

Основные HTTP‑эндпоинты backend описаны в [`backend/README.md`](backend/README.md). Типичный набор:

- `GET /api/health` — проверка здоровья сервера (возвращает статус OK, версию и т.п.);
- `GET /api/devices` — список зарегистрированных устройств (их идентификаторы, базовая информация, последний статус);
- `GET /api/status/:deviceId` — текущий статус конкретного устройства по его ID;
- `POST /api/command/:deviceId` — отправка команды устройству.

Фактический набор эндпоинтов можно уточнить в [`server.js`](backend/src/server.js). Общие принципы:

- все ответы возвращаются в формате JSON;
- в заголовках запросов/ответов может использоваться токен аутентификации пользователя;
- ошибки возвращаются с соответствующими HTTP‑кодами (4xx для ошибок клиента, 5xx для ошибок сервера) и полями `error`/`message` в JSON.

### 3.1 Пример формата команд

Тело `POST /api/command/:deviceId` обычно содержит JSON:

```json
{
  "type": "control",
  "payload": {
    "relay": "pump1",
    "action": "toggle"
  }
}
```

или другой тип команды (например, изменение настроек или запрос на перезагрузку устройства). Backend не реализует саму бизнес‑логику WaterSysPro, а только маршрутизирует команды через MQTT к конкретному ESP32.


## 4. Интеграция с MQTT

### 4.1 Брокер и топики

Для обмена данными используется брокер MQTT Mosquitto, разворачиваемый через [`docker-compose.yml`](docker-compose.yml) с конфигурацией в [`mosquitto/config`](mosquitto/config). Типичная схема топиков:

- `watersys/<deviceId>/status` — статус/телеметрия от устройства к backend;
- `watersys/<deviceId>/command` — команды от backend к устройству;
- дополнительные топики для служебной информации (например, `.../log`, `.../event`).

Фактические имена и формат топиков задаются в прошивке ESP32 (модуль `vqtt`) и настройках backend. Они должны быть согласованы между прошивкой и backend.

### 4.2 Обработка сообщений

Backend подписывается на статусные топики и при получении сообщения:

1. проверяет, что сообщение относится к известному `deviceId`;
2. разбирает JSON‑payload (состояние реле, уровни, состояние FSM, аварийные флаги и т.п.);
3. обновляет запись устройства в базе данных (см. модуль [`db.js`](backend/src/db.js));
4. при необходимости сохраняет историю изменений или генерирует уведомления.

При отправке команд backend формирует JSON‑payload и публикует его в соответствующий `.../command` топик, откуда его забирает ESP32.


## 5. Работа с базой данных

Backend использует базу данных (обычно PostgreSQL) для хранения:

- списка устройств и их привязки к пользователям;
- последнего известного статуса каждого устройства;
- истории событий (по необходимости);
- параметров конфигурации и пользовательских настроек.

Модуль [`db.js`](backend/src/db.js) содержит:

- функции подключения к БД с использованием конфигурации из переменных окружения (см. `.env.example` в [`backend`](backend));
- SQL‑запросы или ORM‑обёртки для операций CRUD с устройствами и событиями;
- инициализацию структуры таблиц (миграции) или описание схемы.

Конкретная схема БД описана в коде и/или README backend, но общая идея — хранить достаточный минимум данных для отображения устройств и их состояния в мобильных/веб‑клиентах.


## 6. Конфигурация и запуск backend

### 6.1 Переменные окружения

Backend настраивается через переменные окружения, перечисленные в файле [`backend/.env.example`](backend/.env.example). Типичный набор:

- параметры подключения к БД: `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`;
- параметры MQTT: `MQTT_HOST`, `MQTT_PORT`, `MQTT_USER`, `MQTT_PASSWORD` (при необходимости);
- сетевые параметры HTTP‑сервера: `PORT`, `HOST`;
- секреты для подписи JWT или других токенов аутентификации.

Для запуска в Docker значения подставляются через `docker-compose.yml`.

### 6.2 Docker‑запуск

Сценарий развёртывания описан в [`README_PoC.md`](README_PoC.md) и [`docker-compose.yml`](docker-compose.yml):

1. Скопировать `.env.example` в `.env` и заполнить значения.
2. Запустить `docker-compose up -d` в корне проекта:
   - поднимается брокер Mosquitto;
   - поднимается база данных (PostgreSQL);
   - поднимается контейнер backend (Node.js) и, при необходимости, Nginx‑прокси.
3. Проверить доступность backend по `http://<host>:<port>/api/health`.


## 7. Интеграция с ESP32 и мобильным приложением

### 7.1 Связка с ESP32

ESP32‑прошивка и backend взаимодействуют через:

- MQTT‑топики (статус/команды);
- (опционально) прямые HTTP‑эндпоинты на ESP32 (`/status`, `/control`, `/api/*`), которые может вызывать backend для отладки или прямого доступа.

Адрес ESP32, MQTT‑брокера и настройки топиков задаются в прошивке (модуль `vqtt` и настройки в `Preferences`). Эти параметры должны соответствовать конфигурации backend и `docker-compose`.

### 7.2 Связка с мобильным приложением

Мобильное приложение использует backend как единый API‑шлюз:

- для получения списка устройств и их статуса — через `GET /api/devices` и `GET /api/status/:deviceId`;
- для управления устройствами — через `POST /api/command/:deviceId`;
- для аутентификации и управления пользователями (если реализовано) — через соответствующие эндпоинты.

Структура ответов backend согласована с типами и интерфейсами в мобильном приложении (см. `services/api.ts` в каталоге [`mobile`](mobile)).


## 8. Рекомендации по расширению backend

При добавлении новых возможностей рекомендуется придерживаться следующих шагов:

1. **Определить новые сценарии использования.**
   - Например: просмотр истории логов, продвинутый мониторинг аварий, массовое управление группами устройств.

2. **Расширить HTTP API.**
   - Добавить новые маршруты в [`server.js`](backend/src/server.js);
   - описать их в [`backend/README.md`](backend/README.md) и общей документации.

3. **Обновить схему БД.**
   - Добавить новые таблицы/поля в [`db.js`](backend/src/db.js) или системе миграций;
   - обеспечить обратную совместимость и миграцию данных.

4. **Согласовать MQTT‑протокол.**
   - При необходимости добавить новые типы сообщений или топики;
   - обновить прошивку ESP32 (модуль `vqtt`) и документацию.

5. **Обновить мобильное приложение и Web‑клиенты.**
   - Добавить использование новых эндпоинтов в `services/api.ts` (см. документацию по мобильному приложению);
   - обновить UI для отображения новых данных и команд.

Следуя этим рекомендациям, можно развивать backend без нарушения существующей интеграции с ESP32 и клиентами.
