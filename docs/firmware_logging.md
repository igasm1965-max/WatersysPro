# Логирование событий и работа с SD в WaterSysPro

## 1. Назначение подсистемы логирования

Подсистема логирования в WaterSysPro регистрирует важные события работы контроллера:

- изменения состояния конечного автомата (FSM);
- срабатывания аварий и защит (в том числе сухой ход, таймауты стадий);
- действия пользователя через локальное меню и Web‑интерфейс;
- сетевые события (WiFi, MQTT);
- служебные операции с SD‑картой, логами и конфигурацией.

Это позволяет:

- диагностировать поведение системы в полевых условиях;
- анализировать причины аварий и некорректной работы;
- отслеживать статистику эксплуатации (частота промывок, перезапусков и т.п.).

Интерфейс логирования определён в заголовке [`event_logging.h`](include/event_logging.h) и реализован в [`event_logging.cpp`](src/event_logging.cpp). Детали интеграции с SD‑картой описаны также в [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md) и модуле [`utils.cpp`](src/utils.cpp).


## 2. Структура записи лога (EventLog)

Базовая структура одной записи журнала определена в [`structures.h`](include/structures.h) как [`EventLog`](include/structures.h:182). Типичная запись содержит:

- дату и время события (год, месяц, день, часы, минуты, секунды);
- тип события `eventType` (код категории события);
- параметр `eventParam` (дополнительный код/значение, зависящий от типа события);
- дополнительную информацию `additionalInfo` (например, источник команды, результат операции);
- уровень важности `level` (`LogLevel`).

Серийный формат строки лога, описанный в [`README.md`](README.md), имеет вид:

```text
DD.MM.YYYY HH:MM:SS - type:XX param:YY lvl:Z - TEXT
```

Где:

- `type` — числовой код события (например, смена состояния, авария, операция пользователя);
- `param` — параметр события (например, новое состояние FSM, код настройки, номер реле);
- `lvl` — уровень логирования (`0` — debug, `1` — info, `2` — warning, `3` — error);
- `TEXT` — текстовое описание, формируемое функциями форматирования.

Уровни логирования перечислены в [`LogLevel`](include/structures.h:25).


## 3. Типы событий и источники

### 3.1 Коды событий

Числовые коды событий определены в [`config.h`](include/config.h) (группа `EVENT_*`). Примеры:

- `EVENT_SYSTEM_START` — старт системы;
- `EVENT_SYSTEM_STATE_CHANGE` — смена состояния FSM;
- `EVENT_UNEXPECTED_STATE_CHANGE` — обнаружен некорректный переход состояния;
- `EVENT_EMERGENCY` — вход в аварийный режим;
- `EVENT_SD_MOUNTED` / `EVENT_SD_ERROR` — события, связанные с SD‑картой;
- события, связанные с WiFi и MQTT, операциями пользователя, мануальным управлением реле и т.д.

Описание событий в человекочитаемом виде обеспечивают функции форматирования в [`event_logging.cpp`](src/event_logging.cpp), например:

- [`getEventTypeShort()`](src/event_logging.cpp:1116) — краткое обозначение события;
- [`getEventTypeLong()`](src/event_logging.cpp:1153) — развернутое описание события;
- [`getStateDescription()`](src/event_logging.cpp:1199) — текстовое описание состояния FSM по его коду;
- [`getSettingDescription()`](src/event_logging.cpp:1217) — описание настройки по её коду;
- [`getManualOperationDescription()`](src/event_logging.cpp:1238) — описание ручной операции (например, включение конкретного реле).

### 3.2 Источник события (LogSource)

Дополнительное поле источника (например, Web, MQTT, кнопка, система) задается перечислением [`LogSource`](include/event_logging.h:13) в [`event_logging.h`](include/event_logging.h). Это позволяет, например, отличать события, вызванные локальным меню, от событий, инициированных удалённо через Web‑API или MQTT.

Поле источника обычно кодируется в `additionalInfo` или параметрах события и используется при форматировании текста.


## 4. API логирования

### 4.1 Основные функции

Основные точки входа для записи событий описаны в [`event_logging.h`](include/event_logging.h) и реализованы в [`event_logging.cpp`](src/event_logging.cpp):

- [`saveEventLog(LogLevel level, uint8_t eventType, uint16_t param, uint16_t additionalInfo)`](src/event_logging.cpp:562)
  - записывает новое событие с указанным уровнем, типом и параметрами;
  - добавляет запись в буфер в памяти и, при необходимости, в файл на SD/SPIFFS;
- перегрузки `saveEventLog` с меньшим числом аргументов (когда дополнительные параметры не нужны или могут быть заданы по умолчанию).

Внутри `saveEventLog`:

1. Выполняется сопоставление (через ламбду `mapEventToBit`) типа и уровня события с маской фильтра (см. раздел 5).
2. При разрешении события маской формируется строка лога с помощью `snprintf` (см. участок кода вокруг [`snprintf`](src/event_logging.cpp:658)).
3. Строка добавляется в файл лога на SD‑карте (если она доступна) или в SPIFFS (fallback).
4. Удаляются слишком старые записи (по необходимости), обновляется буфер последних N событий в памяти.

### 4.2 Вспомогательные функции

Подсистема логирования содержит набор вспомогательных функций для работы с журналом:

- загрузка логов при старте: [`loadEventLogs()`](src/event_logging.cpp:746);
- экспорт логов в текстовый файл на SD: [`exportLogsToSD()`](src/event_logging.cpp:784) и [`exportEventsAsText()`](src/event_logging.cpp:1005);
- очистка всех логов: [`clearEventLogs()`](src/event_logging.cpp:868);
- удаление старых логов старше заданного возраста: [`deleteOldEventLogs()`](src/event_logging.cpp:884), [`pruneEventLogs()`](src/event_logging.cpp:957);
- поиск последнего времени наступления события определённого типа: [`getLastEventTime()`](src/event_logging.cpp:1280);
- фильтрация по коду события: [`filterEventsByCode()`](src/event_logging.cpp:1067).

Эти функции используются как локальным UI (меню и дисплей), так и Web‑интерфейсом (для экспорта логов, формирования статистики и т.п.).


## 5. Фильтрация событий и уровни логирования

### 5.1 Маска фильтра и категории

Чтобы не захламлять логи малозначительными записями, система использует битовую маску фильтрации. В [`config.h`](include/config.h) определены флаги `LOG_FILTER_*`, описывающие категории событий, например:

- `LOG_FILTER_SYSTEM` — системные события (старт, смена состояния, конфигурация);
- `LOG_FILTER_EMERGENCY` — аварийные события;
- `LOG_FILTER_USER` — действия пользователя;
- `LOG_FILTER_NETWORK` — сетевые события (WiFi, MQTT);
- `LOG_FILTER_SD` — операции с SD‑картой;
- и т.д.

Текущее значение маски фильтрации хранится в NVS (ключ, например, `PREF_KEY_LOG_FILTER`) и может изменяться пользователем через инженерное меню или Web‑интерфейс.

### 5.2 Применение фильтра в saveEventLog

Внутри [`saveEventLog()`](src/event_logging.cpp:562) выполняется сопоставление события с маской фильтра. Лямбда `mapEventToBit` (см. участок кода [`mapEventToBit`](src/event_logging.cpp:603)) определяет, к какой категории относится данное событие, и возвращает соответствующий бит. Если этот бит выключен в текущей маске, событие может:

- не записываться в файл лога;
- или записываться только в оперативный буфер (в зависимости от конфигурации).

Таким образом, оператор/инженер может, например, включить только аварийные и системные события, отключив подробный debug‑уровень.


## 6. Хранение логов: SD и SPIFFS

### 6.1 Определение носителя

Функция [`logsOnSD()`](src/event_logging.cpp:14) определяет, куда следует писать логи в текущей конфигурации:

- если SD‑карта успешно инициализирована (флаг `sdPresent` в [`utils.cpp`](src/utils.cpp)), то основной журнал хранится на SD;
- если SD недоступна, используется SPIFFS (на старых версиях прошивки).

Инициализация SD‑карты выполняется в [`initSD()`](src/utils.cpp:71). При успешном монтировании SD устанавливается флаг `sdPresent`, выполняется тестовая запись файла и возможная миграция UI‑файлов через [`migrateUiToSd()`](src/utils.cpp:141). Подробности приведены в [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md).

### 6.2 Формат и размещение файлов

Как правило, логи хранятся в текстовом файле на SD (например, `/events.log`) с одной записью на строку.

Дополнительно могут использоваться вспомогательные файлы:

- экспортированный лог за период в отдельном файле (например, `/exported_events.txt`), создаваемом функцией [`exportLogsToSD()`](src/event_logging.cpp:784);
- файлы статистики или служебные файлы (в том числе для миграции со старых форматов).

### 6.3 Миграция старого формата логов

Для поддержки обновления прошивки предусмотрена миграция старого бинарного формата логов. Логика миграции реализована в функции [`migrateOldBinaryLog()`](src/event_logging.cpp:493):

- при старте система проверяет наличие старых бинарных файлов логов на SD или в SPIFFS;
- если они обнаружены, производится конвертация в новый текстовый формат;
- после успешной миграции старые файлы могут быть удалены или помечены как архивные.

Также предусмотрена инициализация SPIFFS (при необходимости) через [`initSPIFFS()`](src/event_logging.cpp:547).


## 7. Обслуживание логов и управление через UI

### 7.1 Автоматическая очистка

Чтобы логи не занимали всё доступное пространство, реализованы функции автоматической очистки:

- [`deleteOldLogs()`](src/event_logging.cpp:884) и [`deleteOldEventLogs()`](src/event_logging.cpp:884) — удаление записей старше указанного возраста;
- [`pruneEventLogs()`](src/event_logging.cpp:957) — ограничение объёма логов по времени или количеству;
- [`checkAndCleanOldLogs()`](src/event_logging.cpp:1263) — периодическая проверка условий очистки (может вызываться из основного цикла или специализированных задач).

Настройки политики очистки (порог возраста и т.п.) могут задаваться константами или параметрами инженерного меню.

### 7.2 Просмотр логов на дисплее

Модуль логирования содержит функции для просмотра логов на LCD через меню:

- отображение списка событий: [`displayEventList()`](src/event_logging.cpp:1297);
- подробности по конкретному событию: [`displayEventDetails()`](src/event_logging.cpp:1338);
- навигация по журналу и взаимодействие с пользователем: [`eventLogBrowser()`](src/event_logging.cpp:1389).

Эти функции интегрированы с подсистемой меню (см. `docs/firmware_io.md`) и позволяют пользователю листать логи, просматривать подробности, применять фильтры и запускать очистку.

### 7.3 Статистика и графики

Для более глубокого анализа предусмотрены функции:

- отображение статистики событий: [`displayEventStats()`](src/event_logging.cpp:19);
- визуализация графика событий по времени: [`displayEventGraph()`](src/event_logging.cpp:316).

Эти функции используют накопленные данные логов и дают оператору дополнительный взгляд на динамику работы системы.


## 8. Интеграция с Web‑интерфейсом и SD API

Подсистема логирования тесно связана с Web‑интерфейсом и API работы с SD (см. `docs/firmware_web_api.md` и [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md)):

- Web‑интерфейс может загружать лог‑файлы с SD через специальные HTTP‑эндпоинты (описаны в `SD_INTEGRATION.md` и реализованы в [`web_server.cpp`](src/web_server.cpp));
- через Web‑UI и/или API можно инициировать экспорт логов, их очистку, форматирование SD и другие операции обслуживания;
- список файлов на SD и операции над ними (удаление, скачивание) позволяет оператору управлять логами без доступа к локальному меню.

JSON‑ответы Web‑API могут включать агрегированную информацию о логах (например, наличие файла логов, размер, время последнего события). При необходимости можно расширить Web‑API, добавив специализированные эндпоинты для получения последних N событий или статистики по категориям.


## 9. Рекомендации по расширению подсистемы логирования

При расширении или модификации логирования рекомендуется:

1. **Добавлять новые коды событий в одном месте.**
   - Определять новые `EVENT_*` в [`config.h`](include/config.h) с понятными комментариями.

2. **Обновлять человекочитаемые описания.**
   - Добавлять соответствующие ветки в [`getEventTypeShort()`](src/event_logging.cpp:1116), [`getEventTypeLong()`](src/event_logging.cpp:1153), [`getSettingDescription()`](src/event_logging.cpp:1217) и др.

3. **Учитывать фильтрацию.**
   - Определить категорию нового события в `mapEventToBit` (см. [`mapEventToBit`](src/event_logging.cpp:603)) и при необходимости добавить новый флаг `LOG_FILTER_*`.

4. **Сохранять обратную совместимость.**
   - При изменении формата логов реализовывать миграцию в [`migrateOldBinaryLog()`](src/event_logging.cpp:493) или аналогичной функции;
   - по возможности не ломать обработку существующих логов на стороне внешних инструментов.

5. **Минимизировать нагрузку на флеш/SD.**
   - Не логировать слишком частые события на высоком уровне подробности;
   - использовать фильтрацию и ротацию логов;
   - хранить только действительно полезную информацию для диагностики.

Следование этим принципам поможет поддерживать журналы событий информативными и удобными для анализа, не перегружая при этом SD‑карту и интерфейс пользователя.
