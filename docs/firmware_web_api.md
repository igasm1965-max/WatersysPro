# Web‑сервер и HTTP/WebSocket API WaterSysPro

## 1. Обзор Web‑подсистемы

Прошивка WaterSysPro включает асинхронный Web‑сервер и WebSocket‑подсистему для:

- предоставления статуса системы в формате JSON;
- удалённого управления реле и запуском технологических операций;
- настройки сети (WiFi), MQTT и admin‑token;
- работы с файлами на SD‑карте (включая логи и Web‑UI);
- обслуживания простого Web‑интерфейса (SPA и legacy‑страницы).

Основные файлы:

- интерфейс и объявления: [`web_server.h`](include/web_server.h)
- реализация Web‑сервера и API: [`web_server.cpp`](src/web_server.cpp)
- вспомогательные функции и SD‑интеграция: [`utils.cpp`](src/utils.cpp), [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md)
- статические файлы Web‑интерфейса для SD: [`index.html`](закинуть на сд/index.html), [`logfilter.html`](закинуть на сд/logfilter.html)

В качестве HTTP‑стека используется библиотека ESPAsyncWebServer (см. зависимости в [`platformio.ini`](platformio.ini)).


## 2. Архитектура Web‑сервера

### 2.1 Инициализация

Инициализация Web‑сервера выполняется функцией `initWebServer()` в [`web_server.cpp`](src/web_server.cpp:780). В ходе инициализации:

- создаётся глобальный экземпляр `AsyncWebServer server(80)` и `AsyncWebSocket ws("/ws")`;
- регистрируются обработчики HTTP‑эндпоинтов (`/`, `/status`, `/control`, `/api/*`, SD‑эндпоинты и др.);
- настраивается обработчик событий WebSocket `onWsEvent`;
- включается периодическая отправка статуса через `webServerPeriodic()` и мониторинг через `webMonitorPeriodic()`.

### 2.2 Периодическое обслуживание

Функция [`webServerPeriodic()`](src/web_server.cpp:64) вызывается из основного цикла и выполняет:

- трансляцию статуса системы всем подключенным WebSocket‑клиентам с помощью [`broadcastStatus()`](src/web_server.cpp:71);
- вызов `webMonitorPeriodic()` для вывода диагностической информации в `Serial` (статистика памяти, количество подключенных клиентов и т.п.).

Интервалы отправки статуса и диагностики зашиты в коде (`broadcastStatus` — примерно раз в 2 секунды, `webMonitorPeriodic` — реже).


## 3. Формирование статуса системы

### 3.1 Эндпоинт /status

HTTP‑эндпоинт `/status` обслуживается функцией [`handleStatus()`](src/web_server.cpp:247). Она формирует JSON‑объект, содержащий текущий статус системы, и отправляет его в ответ на GET‑запрос.

Пример структуры ответа (поля могут меняться по мере развития прошивки):

- `ip` — IP‑адрес устройства в сети;
- `mac` — MAC‑адрес устройства (форматируется через [`getMacString()`](src/utils.cpp:331));
- `uptime` — время работы с момента перезапуска (в секундах);
- `state` — текстовое обозначение текущего состояния FSM (через `getStateText`/`getStateDescription`);
- `state_code` — числовой код состояния (`SystemState`);
- `tank1` / `tank2` — уровни воды в баках;
- `tank1Empty` / `tank2Empty` — логические флаги пустоты баков;
- `filterCleaningNeeded` — флаг необходимости обратной промывки фильтра;
- `backwashInProgress` — флаг активной обратной промывки;
- `timers` — объект с оставшимся временем стадий (озонация, аэрация, отстаивание, обратная промывка);
- `relays` — объект с состояниями реле (pump1, pump2, aeration, ozone, filter, backwash);
- `sdPresent` — наличие SD‑карты;
- `mqtt` — объект со статусом MQTT (включено/выключено, подключено/нет, broker, topic base);
- `emergency` — объект с информацией об аварийном режиме (флаг, текст сообщения).

Этот JSON используется как локальной Web‑страницей (`index.html`), так и внешними клиентами (backend, мобильное приложение).

### 3.2 WebSocket‑трансляция статуса

Функция [`broadcastStatus()`](src/web_server.cpp:71) формирует JSON, аналогичный `/status`, и рассылает его всем подключенным WebSocket‑клиентам через `ws.textAll(...)`. Она вызывается из `webServerPeriodic()` с ограничением по частоте.

Функция [`sendStatusNow()`](src/web_server.cpp:147) формирует и отправляет статус немедленно (используется, например, после выполнения команды управления, чтобы клиент сразу увидел результат).


## 4. Управление реле и режимами через Web‑API

### 4.1 HTTP‑эндпоинт /control

Эндпоинт `/control` (обычно метод GET или POST) обрабатывается функцией [`handleControl()`](src/web_server.cpp:324). Он предназначен для удалённого управления реле и, косвенно, технологическими стадиями.

Типичный формат запроса:

- query‑параметр `action` или `relay`, задающий, какое устройство нужно переключить (например, `pump1`, `pump2`, `aeration`, `ozone`, `filter`, `backwash`);
- возможные дополнительные параметры (режим on/off или toggle).

Обработчик:

1. Валидирует запрос (например, наличие параметра, проверка admin‑token, если требуется).
2. Вызывает соответствующие функции из модуля [`relay_control`](src/relay_control.cpp) или вспомогательные функции `setRelayState`.
3. Логирует событие как ручную операцию (`EVENT_MANUAL_OPERATION`) с источником `SRC_WEB` (см. [`LogSource`](include/event_logging.h:13)).
4. Возвращает JSON‑ответ с полем `success` и при необходимости вызывает [`sendStatusNow()`](src/web_server.cpp:147).

### 4.2 WebSocket‑сообщения управления

Обработчик событий WebSocket [`onWsEvent()`](src/web_server.cpp:457) умеет принимать JSON‑сообщения от клиента. Наиболее важные типы сообщений:

- `{"type":"control", ...}` — команды управления реле и режимами, аналогичные `/control`;
- `{"type":"wifi", ...}` — настройка WiFi (см. раздел 5.2);
- возможны другие типы для служебных операций.

При получении сообщения управления:

1. JSON парсится с помощью ArduinoJson;
2. по полю `type` выбирается режим обработки;
3. выполняются те же действия, что и в `/control` (включая логирование `EVENT_MANUAL_OPERATION`);
4. клиенту отправляется подтверждение и/или обновлённый статус через WebSocket.

Использование WebSocket для управления позволяет реализовать более отзывчивый UI без постоянных HTTP‑запросов.


## 5. Настройка WiFi, admin‑token и MQTT через API

### 5.1 HTTP‑эндпоинт /api/wifi

Эндпоинт `POST /api/wifi` обрабатывается функцией [`handlePostWifiBody()`](src/web_server.cpp:575), которая принимает JSON‑тело запроса:

- `ssid` — имя WiFi‑сети для подключения в режиме STA;
- `password` — пароль WiFi (может быть пустым для открытых сетей, но в реальных условиях это нежелательно);
- `token` — текущий admin‑token (для авторизации операции);
- дополнительные поля (например, `lang_en` для настройки языка интерфейса).

При обработке запроса:

1. Проверяется корректность JSON и наличие обязательных полей;
2. Достаётся текущий admin‑token из NVS (через `Preferences` и ключи, определённые в [`config.h`](include/config.h));
3. Проводится сравнение полученного токена с сохранённым (безопасность);
4. При успехе новые значения `ssid` и `password` сохраняются в NVS;
5. Формируется JSON‑ответ с признаком успеха/ошибки и, при необходимости, сообщением для пользователя.

### 5.2 HTTP‑эндпоинт /api/admin_token

Эндпоинт `POST /api/admin_token` обрабатывается функцией [`handlePostAdminTokenBody()`](src/web_server.cpp:628). JSON‑тело содержит:

- `old_token` — текущий admin‑token (для проверки полномочий);
- `new_token` — желаемый новый admin‑token.

Правила и ограничения:

- длина нового токена ограничена константой `ADMIN_TOKEN_MAX_LEN` из [`config.h`](include/config.h);
- допускаются только определённые символы (обычно буквенно‑цифровые), проверка выполняется в коде обработчика;
- при успешной проверке токен сохраняется в NVS.

Admin‑token используется для защиты критически важных операций, таких как настройка WiFi, MQTT, управление файлами на SD и доступ к инженерным функциям через Web‑интерфейс.

### 5.3 HTTP‑эндпоинты /api/mqtt

Для настройки MQTT предусмотрены эндпоинты:

- `GET /api/mqtt` — обработчик [`handleGetMqtt()`](src/web_server.cpp:710) возвращает текущие настройки MQTT (broker, порт, topic base, флаги включения и т.п.);
- `POST /api/mqtt` — обработчик [`handlePostMqttBody()`](src/web_server.cpp:727) принимает JSON‑тело с новыми параметрами MQTT и сохраняет их в NVS.  Ранее требовалась проверка admin‑token, но сейчас настройки можно менять без него.

Эти настройки используются модулем [`vqtt`](src/vqtt.cpp) и [`wifi_manager`](src/wifi_manager.cpp) для подключения к брокеру и публикации/подписки на топики.


## 6. Работа с файлами на SD‑карте и логами через Web‑API

### 6.1 Обзор SD‑эндпоинтов

Подробное описание SD‑эндпоинтов приведено в [`SD_INTEGRATION.md`](docs/SD_INTEGRATION.md). Ключевые операции:

- получение списка файлов на SD;
- скачивание отдельного файла (например, логов);
- удаление файлов (очистка логов, удаление старых UI‑файлов и т.п.);
- (опционально) форматирование SD‑карты.

Реализация этих эндпоинтов находится в нижней части [`web_server.cpp`](src/web_server.cpp) и использует:

- функции доступа к SD и мьютекс `sdMux` из [`utils.cpp`](src/utils.cpp);
- проверки admin‑token для операций удаления/форматирования.

### 6.2 Пример: получение списка файлов

По описанию из `SD_INTEGRATION.md` Web‑сервер предоставляет эндпоинт (например, `GET /api/sd_files`), который:

- сканирует корень SD‑карты или выделенную директорию;
- формирует JSON‑массив с объектами вида `{ "name": "filename", "size": 12345, "mtime": ... }`;
- возвращает его клиенту.

UI `index.html` может использовать этот список для отображения доступных логов и других файлов.

### 6.3 Удаление и скачивание файлов

Другой эндпоинт (например, `GET /api/sd_file?name=...`) позволяет:

- скачать файл по имени в виде HTTP‑ответа с соответствующим `Content-Type`;
- предварительно проверить наличие файла и права доступа.

Удаление файлов реализовано, как правило, через `POST /api/sd_files` с JSON‑телом вида `{ "action": "delete", "name": "events.log", "token": "..." }`:

1. Проверяется токен (admin‑token);
2. Пытается удалиться указанный файл или файлы по маске;
3. Возвращается JSON с результатом (успех/ошибка и пояснение).

Таким образом, оператор может управлять логами и другими файлами, не подключаясь к устройству по Serial и не вынимая SD‑карту.


## 7. Web‑интерфейс: SPA и legacy UI

### 7.1 Статические ресурсы

Обслуживание статических файлов реализовано в функции [`sendStaticResource()`](src/web_server.cpp:46), которая:

- сначала пытается найти запрошенный файл на SD‑карте;
- при отсутствии — может использовать SPIFFS (на старых версиях прошивки);
- устанавливает корректный `Content-Type` по расширению файла.

Главная страница (`/`) обрабатывается функцией [`handleRoot()`](src/web_server.cpp:236), которая обычно вызывает `sendStaticResource` для `index.html`.

### 7.2 SPA‑интерфейс

Файл [`index.html`](закинуть на сд/index.html) реализует одностраничное приложение (SPA), которое:

- регулярно запрашивает `/status` или подписывается на WebSocket для получения статуса;
- отображает уровни баков, состояние FSM, состояние реле и аварий;
- предоставляет кнопки управления (включение/отключение насосов, запуск промывки, очистка логов и т.п.), которые отправляют запросы на `/control` или WebSocket.

Дополнительная страница [`logfilter.html`](закинуть на сд/logfilter.html) может использоваться для работы с фильтрами логов и просмотра/скачивания лог‑файлов.

### 7.3 Legacy‑интерфейс

Для совместимости и отладки предусмотрен простой legacy‑интерфейс, обрабатываемый функцией [`handleLegacy()`](src/web_server.cpp:411). Он представляет собой базовую HTML‑страницу с ссылками и формами для:

- отображения состояний реле и их ручного управления;
- доступа к файлам логов и другим файлам на SD;
- выполнения некоторых сервисных операций.

Этот интерфейс полезен, когда SPA ещё не загружен на SD или находится в разработке.


## 8. Безопасность и авторизация

### 8.1 Admin‑token

Admin‑token — это строка, используемая для авторизации критических операций. Параметры токена:

- максимальная длина `ADMIN_TOKEN_MAX_LEN` определена в [`config.h`](include/config.h);
- токен хранится в NVS под специальным ключом (например, `PREF_KEY_ADMIN_TOKEN`);
- токен используется в эндпоинтах `/api/wifi`, `/api/admin_token`, `/api/mqtt`, SD‑эндпоинтах и, возможно, других.

Проверка токена выполняется в обработчиках `handlePostWifiBody`, `handlePostAdminTokenBody`, `handlePostMqttBody` и SD‑эндпоинтах: запросы без правильного токена отклоняются.

### 8.2 Ограничение операций

Не все операции требуют токен. Типичная модель:

- чтение статуса (`/status`, WebSocket‑статус) — без токена;
- простые операции управления реле в локальной сети — могут быть без токена (в зависимости от конфигурации);
- изменение настроек (WiFi, MQTT, уровни, таймауты) и операции с SD (удаление, форматирование) — только с валидным admin‑token;
- изменение самого admin‑token — только при предъявлении текущего токена.

Эту модель при необходимости можно ужесточить (например, требовать токен для всех команд управления).

### 8.3 Рекомендации по безопасности

- всегда менять значение admin‑token по умолчанию сразу после установки системы;
- не передавать токен в URL (query‑параметры) при удалённом доступе через интернет;
- использовать HTTPS‑прокси и VPN/туннели при доступе из внешней сети (на уровне инфраструктуры, не контроллера);
- по возможности ограничивать доступ к Web‑интерфейсу только локальной сетью.


## 9. Расширение Web‑API

При добавлении новых функций в Web‑интерфейс или интеграции с внешними системами можно расширять Web‑API. Рекомендуемый процесс:

1. **Определить модель данных и формат JSON.**
   - Какие поля будут входить в запрос/ответ;
   - Как они соотносятся с внутренними структурами (`SystemFlags`, `SafetySettings`, FSM и т.п.).

2. **Добавить эндпоинт в `web_server.cpp`.**
   - Зарегистрировать новый обработчик в `initWebServer()`;
   - Реализовать логику парсинга запроса и формирования ответа.

3. **Учесть авторизацию.**
   - Решить, нужен ли admin‑token для нового эндпоинта;
   - Добавить проверку токена по аналогии с существующими обработчиками.

4. **Взаимодействие с остальными модулями.**
   - Использовать уже существующие API (`relay_control`, `prefs`, `event_logging`, FSM), а не дублировать логику;
   - При необходимости добавить новые функции в соответствующие модули.

5. **Обновить UI и документацию.**
   - Добавить поддержку нового эндпоинта в `index.html` и/или мобильное приложение;
   - Описать эндпоинт в этом документе и, при необходимости, в общей документации по системе.

Следуя этим рекомендациям, можно постепенно развивать Web‑интерфейс и интеграцию, не нарушая существующую архитектуру и безопасность системы.
