# Поддержка microSD (SPI) — интеграция

Дата: 19 февраля 2026 г.
Автор: GitHub Copilot

## Кратко
Добавлена начальная поддержка microSD в SPI‑режиме:
- Инициализация `initSD()` и вывод статуса в Serial
- Пины по умолчанию: SCK=GPIO4, MISO=GPIO19, MOSI=GPIO23, CS=GPIO33, CD=GPIO39 (опционально)
- В `setup()` вызывается `initSD()` — если карта не подключена, система продолжает работать и использует SPIFFS как fallback

## Что сделано в коде
- `include/config.h` — добавлены defines для SD пинов
- `include/utils.h` — объявлена `bool initSD();`
- `src/utils.cpp` — реализована `initSD()` (SPI); объект `SPIClass` теперь статический, иначе локальный экземпляр разрушался и приводил к сбоям при последующих вызовах
- `src/main.cpp` — вызов `initSD()` в `setup()` и образец простого теста R/W
- `src/main.cpp` — вызов `initSD()` в `setup()`
- Serial вывод показывает результат монтирования (успех/неудача)

## Схема подключения (рекомендованная, SPI mode)
- SD VCC  → 3.3V (обязательно)
- SD GND  → GND
- SD MOSI → GPIO23
- SD MISO → GPIO19
- SD SCK  → GPIO4
- SD CS   → GPIO33
- (опц.) SD CD → GPIO39 (input only)

> Внимание: не подключайте VCC к 5V; если модуль содержит level‑shifter/регулятор — проверьте документацию модуля.

## Как проверить (после физического подключения)
1. Подключите карту по указанной разводке.  
2. Перезагрузите устройство и откройте Serial (115200).  
3. В логах должны появиться строки:
   - `[SD] Initializing (SPI)...`
   - `[SD] mounted, cardType=X` и `[SD] test file written` — если карта обнаружена и доступна.

Дополнительно можно посмотреть примерный код в `esp32_poc/sd_rw_example.ino` — это минимальный скетч, который монтирует карту, записывает `/example.txt` и сразу читает содержимое.

## Возможность освобождения флеш‑памяти

После добавления поддержки SD появилась простой способ «разгрузить» внутреннюю
SPIFFS‑память: перенести статические страницы интерфейса и другие ресурсы на карту.
В последней версии кода реализована вспомогательная функция `migrateUiToSd()` которая
автоматически копирует `/index.html` и `/logfilter.html` на карту при её обнаружении.

Обслуживание выполняется через новый хелпер веб‑сервера — любой запрос на эти
ресурсы сначала пытается загрузить файл с SD, затем (если нет карты) из SPIFFS, и
только в крайнем случае возвращается 404.  Перенос UI может сэкономить десятки
килобайт внутренней флеш‑памяти, освобождая её для кода.

Дополнительно к базовым возможностям управления логами добавлены новые API‑эндпойнты:

* `GET /api/sd_files` — возвращает JSON‑массив строк с именами файлов на карте.
* `GET /api/sd_file?name=<имя>` — скачивает указанный файл (защищён от `..`).
* `POST /api/sd_files` с телом `{token:<админ>,action:"delete",name:"..."}` —
  удаляет файл (нужен админ‑токен).

SPA интерфейс `/` теперь умеет перечислять, скачивать и удалять файлы через эти
эндпойнты; Legacy UI показывает небольшие ссылки.  Кроме того, ответ `/status`
и websocket‑брокасты теперь содержат поле `sd` (0/1) — клиент отображает
индикатор наличия карты.  Это облегчает управление данными прямо из браузера.

Чтобы окончательно отказаться от SPIFFS, достаточно после копирования удалить
файлы из папки `data/` и собрать прошивку в режиме «SD-only».  В `platformio.ini`
есть дополнительная среда `env:esp32dev-sdonly`:

```ini
[env:esp32dev-sdonly]
extends = env:esp32dev
board_build.filesystem = none      # не создаём образ SPIFFS
build_flags = -DUSE_SPIFFS=0       # отключаем код в прошивке
```

Затем выполняйте `platformio run -e esp32dev-sdonly` (или `-t upload` для
прошивки).  Образ SPIFFS не будет загружаться, а компиляция исключит всю
поддержку SPIFFS из кода.

Перед тем как переключаться, убедитесь, что все нужные файлы (UI, логи и т.п.)
скопированы на карту, иначе устройство не сможет их найти.

## Предложения для дальнейшей работы (реализация в следующей итерации)
1. Логирование на SD
   - Буферная запись (batched flush) аналогично текущему SPIFFS buffer
   - Ротация/архивация старых логов
   - API `/api/logs` расширить: `source: sd|spiffs`, `download`/`delete`/`rotate`
2. Файловая система и проверка целостности
   - Проверка свободного места и автоматическое переключение на SPIFFS при переполнении
   - Функция форматирования SD (через Web UI) + проверка ошибок монтирования
3. UI/Backend
   - Добавить индикатор наличия SD в `/status` и в Web UI
   - Кнопки скачивания логов с SD (Legacy UI + SPA)
4. Надёжность и безопасность
   - Блокировки (mutex) для записи/чтения файлов
   - Atomic file replace / safe-rename для избежания corruption
   - Ограничение прав на удаление/форматирование через admin token
5. Тестирование
   - Unit/Integration tests: симуляция отсутствия/вставки карты, тест записи/чтения
   - Smoke‑tests после интеграции (mount, write/read, API)

## Оценка выгоды
- Перенос 1–10 MB логов приведёт к заметному снижению использования внутренней флеш (см. CHANGELOG/notes) — рекомендуется переносить архивы/старые логи и большие файлы.
