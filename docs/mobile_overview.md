# Мобильное приложение WaterSysPro

## 1. Назначение мобильного приложения

Мобильное приложение WaterSysPro предоставляет пользователю удобный интерфейс для:

- просмотра состояния системы подготовки воды (уровни баков, состояние FSM, реле, аварии);
- удалённого управления устройствами (включение/выключение насосов, запуск промывки и т.п.);
- изменения некоторых настроек через backend;
- получения уведомлений о важных событиях (при наличии push‑интеграции).

Приложение реализовано на базе React Native (Expo) и использует HTTP/WebSocket API backend. Исходники приложения находятся в каталоге [`mobile`](mobile).


## 2. Архитектура мобильного приложения

### 2.1 Основные файлы и структура

Ключевые файлы в каталоге [`mobile`](mobile):

- точка входа приложения: [`App.tsx`](mobile/App.tsx)
- конфигурация проекта Expo: [`app.json`](mobile/app.json)
- экранные компоненты:
  - экран входа: [`LoginScreen.tsx`](mobile/src/screens/LoginScreen.tsx)
  - главный экран (dashboard): [`DashboardScreen.tsx`](mobile/src/screens/DashboardScreen.tsx)
  - экран настроек: [`SettingsScreen.tsx`](mobile/src/screens/SettingsScreen.tsx)
- сервисы для работы с API и WebSocket:
  - HTTP API: [`api.ts`](mobile/src/services/api.ts)
  - WebSocket‑клиент: [`websocket.ts`](mobile/src/services/websocket.ts)

Дополнительно в проекте могут присутствовать компоненты UI, утилиты и типы (interfaces) для описания данных, возвращаемых backend.

### 2.2 Навигация и состояние

Приложение использует систему навигации (например, React Navigation), описанную в [`App.tsx`](mobile/App.tsx):

- начальный экран — `LoginScreen`, если пользователь ещё не аутентифицирован;
- после входа пользователь попадает на `DashboardScreen` с основными показателями;
- из dashboard можно перейти на `SettingsScreen` и, при необходимости, на другие вспомогательные экраны (например, подробные логи, если реализованы).

Глобальное состояние (текущий пользователь, выбранное устройство, кэш последнего статуса) хранится либо в контексте React, либо в лёгком состоянии компонентов, с использованием сервисов `api.ts` и `websocket.ts` для синхронизации с backend.


## 3. Связь с backend: HTTP API

### 3.1 Сервис api.ts

Файл [`api.ts`](mobile/src/services/api.ts) инкапсулирует работу с HTTP API backend:

- настройка базового URL backend (IP/порт), который берётся из конфигурации или переменных окружения (например, через `app.json` или dotenv);
- функции для вызова основных эндпоинтов:
  - `login` — аутентификация пользователя (если реализована);
  - `getDevices` — получение списка устройств (`GET /api/devices`);
  - `getDeviceStatus(deviceId)` — получение статуса устройства (`GET /api/status/:deviceId`);
  - `sendCommand(deviceId, command)` — отправка команды устройству (`POST /api/command/:deviceId`).

Каждая функция:

1. формирует HTTP‑запрос к backend;
2. обрабатывает JSON‑ответ;
3. возвращает данные в удобном для компонентов формате или выбрасывает исключение при ошибке.

### 3.2 Конфигурация адресов

Адрес backend и WebSocket сервера может задаваться:

- в `app.json` (поле с IP‑адресом сервера);
- через переменные среды Expo (например, `EXPO_PUBLIC_API_URL`);
- непосредственно в файлах `api.ts` и `websocket.ts` (для отладки).

Важно, чтобы мобильное устройство и backend находились в одной сети или backend был доступен по внешнему адресу.


## 4. Связь с backend: WebSocket

### 4.1 Сервис websocket.ts

Файл [`websocket.ts`](mobile/src/services/websocket.ts) реализует клиентскую сторону WebSocket‑соединения с backend для:

- подписки на обновления статусов устройств в реальном времени;
- получения уведомлений об авариях и других событиях;
- (опционально) отправки команд через WebSocket.

Типичный сценарий:

1. При входе в приложение и выборе устройства создаётся WebSocket‑подключение к backend (например, `ws://<host>/ws`).
2. Приложение отправляет сообщение аутентификации/подписки (при необходимости) с указанием `deviceId`.
3. Backend пересылает через WebSocket обновления статуса (часто используя тот же формат, что и HTTP‑эндпоинт `/status`).
4. Компоненты React подписываются на поток сообщений и обновляют UI.

### 4.2 Управление жизненным циклом WebSocket

Сервис `websocket.ts` должен обрабатывать:

- автоматическое переподключение при обрыве связи;
- таймауты и heartbeat‑сообщения (ping/pong), если требуется;
- отписку и закрытие соединения при выходе пользователя или смене устройства.

Компоненты уровнем выше (например, `DashboardScreen`) должны корректно подписываться и отписываться от событий, чтобы избежать утечек и двойных подписок.


## 5. Экран входа (LoginScreen)

Экран [`LoginScreen.tsx`](mobile/src/screens/LoginScreen.tsx) отвечает за:

- ввод пользователем логина/пароля или другого вида учётных данных;
- вызов функции `login` из `api.ts` и получение токена аутентификации;
- сохранение токена (например, в состоянии или безопасном хранилище);
- переход к `DashboardScreen` при успешном входе;
- отображение ошибок при некорректных данных или недоступности backend.

Тип аутентификации (JWT, сессии, простые токены) определяется реализацией backend.


## 6. Главный экран (DashboardScreen)

Экран [`DashboardScreen.tsx`](mobile/src/screens/DashboardScreen.tsx) — центральное место взаимодействия пользователя с устройством. Основной функционал:

- отображение текущего статуса:
  - уровни воды в баках;
  - состояние FSM и код состояния;
  - состояния реле (pump1, pump2, aeration, ozone, filter, backwash);
  - наличие аварий и их описание;
  - наличие SD‑карты, статус MQTT и др. поля, получаемые от backend;
- отображение списка доступных устройств (если их несколько) и выбор активного;
- элементы управления:
  - кнопки включения/выключения отдельных реле;
  - команды запуска/остановки технологического цикла или обратной промывки;
  - переход к экрану настроек или логов.

Экран периодически обновляет данные:

- через HTTP‑запросы `getDeviceStatus` (polling) и/или
- через подписку на события WebSocket.


## 7. Экран настроек (SettingsScreen)

Экран [`SettingsScreen.tsx`](mobile/src/screens/SettingsScreen.tsx) позволяет пользователю:

- изменять настройки подключения к backend (адрес сервера, порты);
- выбирать язык интерфейса (если поддерживается);
- управлять параметрами уведомлений;
- просматривать и, при наличии прав, менять настройки устройства через backend (например, расписание, параметры безопасности, MQTT‑настройки).

Взаимодействие с backend происходит через функции `api.ts`. При изменении критичных параметров может потребоваться ввод admin‑token или подтверждение операции.


## 8. Сборка и запуск мобильного приложения

### 8.1 Локальный запуск (Expo)

Для разработки и тестирования используется Expo. Общий сценарий:

1. Установить зависимости в каталоге [`mobile`](mobile):
   - `npm install` или `yarn`;
2. Запустить приложение:
   - `npm start` или `expo start`;
3. В Expo‑CLI указать адрес backend (например, в `.env` или через интерфейс приложения);
4. Запустить Expo Go на телефоне или эмуляторе, отсканировать QR‑код и открыть приложение.

Важно убедиться, что мобильное устройство имеет доступ к backend (WiFi‑сеть, VPN и т.п.).

### 8.2 Сборка production‑версий

Сборка production‑версий (APK/AAB для Android, IPA для iOS) выполняется стандартными средствами Expo (EAS Build) и зависит от настроек в [`app.json`](mobile/app.json) и учётной записи Expo.


## 9. Рекомендации по развитию мобильного приложения

При дальнейшей разработке мобильного приложения рекомендуется:

1. **Синхронизировать контракт с backend.**
   - Обновлять типы и интерфейсы данных в `api.ts` при изменении HTTP API backend;
   - следить за совместимостью формата JSON ответов (`/api/status/:deviceId`, `/api/devices`, `/api/command/:deviceId`).

2. **Расширять UI постепенно.**
   - Сначала добавить просмотр истории логов (через backend);
   - затем — управление расписанием, настройками безопасности, SD‑операциями и др.

3. **Оптимизировать трафик и энергорасход.**
   - Использовать WebSocket для push‑обновлений вместо частого polling;
   - уменьшать частоту запросов при неактивном экране.

4. **Обеспечить отказоустойчивость.**
   - Обрабатывать ошибки сети и показывать пользователю понятные сообщения;
   - реализовать повтор запросов при временных ошибках.

Это позволит сохранить согласованность между мобильным приложением, backend и прошивкой ESP32, а также улучшать пользовательский опыт без поломки существующей функциональности.
